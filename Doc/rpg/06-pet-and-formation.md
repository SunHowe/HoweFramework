# 06 · 召唤兽与阵法

> 本章沉淀回合制 RPG 的召唤兽 + 阵法系统设计 —— 不含具体游戏数值,只沉淀"为什么这样设计"。

---

# 第 1 部分:召唤兽(宠物)系统

## 1.1 召唤兽的本质

### 1.1.1 召唤兽是"第二个角色"

召唤兽不是"装备的附属",是"独立角色":
- 独立属性(HP / MP / 攻击 / 防御 / 速度)
- 独立技能(自带技能 + 玩家学习技能)
- 独立 AI(行为树 vs 玩家手动控制)
- 独立装备 / 坐骑 / 饰品

### 1.1.2 召唤兽 vs 召唤物

| 类型 | 描述 | 典型 |
|------|------|------|
| **召唤兽**(宠物 / 宝宝)| 独立角色,持久存在 | 梦幻西游召唤兽、口袋妖怪 |
| **召唤物**(召唤物 / 精灵)| 临时召唤,战斗内存在 | 召唤元素(法师)|

### 1.1.3 召唤兽的设计目标

1. **多样化** —— 不同召唤兽有不同的"长板 / 短板"。
2. **可培养** —— 玩家投入资源能感知成长。
3. **差异化** —— 战斗内有决策空间(参战 / 待机 / 召唤谁)。
4. **长期养成** —— 跨战斗持续,提供长期动力。

## 1.2 召唤兽的 4 大要素(标准)

| 要素 | 描述 | 影响 |
|------|------|------|
| **资质** | 每项属性的"理论上限" | 决定属性成长空间 |
| **成长** | 每次升级的属性成长率 | 决定实际数值 |
| **技能** | 召唤兽自带的技能 + 玩家学技能 | 决定战术 |
| **内丹** | 增强技能的道具 / 装备 | 决定技能强度 |

### 1.2.1 资质(理论上限)

- 每只召唤兽有 6 项资质(攻击 / 防御 / HP / MP / 速度 / 法力 等)
- 资质分 N 档(常见 21 档)
- 最高资质 = 最低资质 × 倍率(常见 1.2)
- 例:某召唤兽攻资最高 1440、最低 1200

### 1.2.2 成长(实际成长率)

- 每次升级,属性按"成长率 × 资质"成长
- 成长率常见 5 档(低 / 中低 / 中 / 中高 / 高)
- 成长率越高,实际数值越高

### 1.2.3 技能

| 类型 | 描述 |
|------|------|
| **自带技能** | 召唤兽天生拥有的技能(1-6 个)|
| **学习技能** | 玩家用技能书给召唤兽学的技能 |
| **技能格** | 召唤兽最多能学的技能数(常见 1-6 格) |

### 1.2.4 内丹

- 内丹 = 增强技能的"装备"
- 高级内丹 vs 低级内丹
- 内丹分 N 层(常见 1-5 层),需要"领悟"升级

## 1.3 召唤兽数据结构

```csharp
public class PetData {
    public int PetId;            // 唯一 id
```

## 1.4 召唤兽的"参战副本"

### 1.4.1 为什么需要"参战副本"

召唤兽的"本体"在玩家背包,战斗中创建的是"参战副本":
- 本体数据不变(等级 / 经验 / 技能)
- 副本只持当前 HP / MP / 状态
- 战斗结束 → 副本销毁,HP/MP/状态保留回本体

### 1.4.2 参战副本的实现

```csharp
// 进入战斗时
var petEntity = CreateEntity();  // 战斗实体
petEntity.AddComponent<TransformComponent>();
petEntity.AddComponent<NumericComponent>();
// 初始化属性 = 本体属性
petEntity.GetComponent<NumericComponent>()
    .Set(NumericId.攻击, NumericSubType.Basic, petInstance.GetAttackAtLevel(level));
// ...

// 战斗结束
var hpLeft = petEntity.GetComponent<ResourceComponent>().Get(ResourceId.HP);
petInstance.Hp = hpLeft;  // 同步回本体
```

## 1.5 召唤兽的"参战规则"

### 1.5.1 参战人数限制

- 常见:玩家每人 1 宠(单人 / 队伍 / 副本不同)
- 召唤兽占一个"队伍位置",影响队伍总数

### 1.5.2 召唤 / 召回

| 操作 | 描述 |
|------|------|
| **召唤** | 把召唤兽加入战斗(消耗 MP 或 道具) |
| **召回** | 召唤兽退出战斗(回血 / 不保留) |
| **参战** | 召唤兽在战斗内行动 |
| **待机** | 召唤兽在场但不出手 |

### 1.5.3 召唤兽死亡

- 召唤兽 HP=0 → 死亡(详见 `08-rules-and-edge-cases.md`)
- 当回合不可再出战
- 战斗结束 → 复活 / 不可复活

## 1.6 召唤兽的 AI

### 1.6.1 手动控制 vs 自动 AI

| 控制方式 | 适用 |
|---------|------|
| **手动控制** | 玩家每回合给召唤兽下指令 |
| **半自动** | 玩家选"跟随 / 主动 / 防御 / 集火"模式 |
| **完全 AI** | 召唤兽自动行为 |

### 1.6.2 AI 决策树(典型)

```
Root (Selector)
  ├─ Sequence: 残血(HP<30%)
  │     ├─ Condition: HP < 30%
  │     ├─ Action: 加血技能
  │     └─ Action: 防御
  ├─ Sequence: 主人危险(主人 HP<50%)
  │     ├─ Condition: 主人 HP < 50%
  │     ├─ Action: 保护主人
  │     └─ Action: 加血技能
  └─ Sequence: 默认
        ├─ Action: 选血最少敌人
        └─ Action: 选最强技能
```

## 1.7 召唤兽的"内丹"

### 1.7.1 内丹 = 技能增强

- 内丹给技能增加"层数"(1-5 层)
- 5 层 + 满成长坐骑统御 ≈ 1 层无统御的 3 倍
- 内丹需要"领悟"才能升级

### 1.7.2 内丹的格数

- 内丹格数由参战等级决定(1-6 格)
- 高级内丹 vs 低级内丹(高级更强)
- 内丹分 1-5 层(初始 1 层)

## 1.8 召唤兽系统的常见边界

### 1.8.1 召唤兽 vs 玩家角色

| 维度 | 召唤兽 | 玩家角色 |
|------|--------|---------|
| 升级 | 自动(战斗后获经验)| 手动(任务 / 战斗)|
| 装备 | 召唤兽专用装备 | 玩家装备 |
| 技能 | 自带 + 学习 | 升级学 |
| 死亡 | 当回合不可出战 | 倒地(可救起)|
| 永久死亡 | 可能(部分游戏)| 不会 |

### 1.8.2 召唤兽的"稀有度"

| 稀有度 | 描述 |
|--------|------|
| **普通** | 常见 |
| **高级** | 稀有 |
| **神兽** | 极其稀有(0.001% 概率)|
| **变异** | 特殊(罕见 / 限时)|

# 第 2 部分:阵法系统

## 2.1 阵法的本质

### 2.1.1 阵法 = 站位 + 属性加成 + 相克

阵法的核心是"站位有属性加成":
- 5 人队伍,每人占一个"阵位"
- 不同阵位 + 不同阵法 → 不同属性加成
- 不同阵法之间有"相克表"

### 2.1.2 阵法的设计目标

1. **团队协作** —— 玩家组队有差异化收益。
2. **站位策略** —— 站位有"最优解"但也"有解"。
3. **阵法相克** —— 增加"知己知彼"的决策维度。
4. **长期养成** —— 阵法可升级 / 替换。

## 2.2 阵法的核心要素

| 要素 | 描述 |
|------|------|
| **阵法列表** | 11 种阵法(常见) |
| **学习上限** | 玩家最多学 N 种阵法(常见 4 种) |
| **生效条件** | 5 人队 + 队长设置 |
| **阵法等级** | 1-5 级 |
| **阵法站位** | 1-5 号位的差异化加成 |
| **阵法相克** | 11×11 矩阵 |

## 2.3 阵法 vs 站位

| 维度 | 阵法 | 站位 |
|------|------|------|
| 描述 | 一组人 + 一组加成 | 单个人的位置 |
| 设计意图 | 队伍层级 | 个人层级 |
| 影响 | 整个队伍的属性 | 单个人的站位 |
| 例子 | 天覆阵、地载阵、风扬阵 | 1 号位 / 2 号位 / 3 号位 |

## 2.4 阵法的站位差异化

### 2.4.1 标准模式

- 1 号位(队长位):差异化加成
- 2-5 号位:差异化加成(每个位置不同)
- 同一阵法对不同位置 → 不同加成

### 2.4.2 站位差异化的设计意图

| 维度 | 设计意图 |
|------|---------|
| **位置差异** | 不同位置有不同加成 → 玩家要考虑站位 |
| **速度差异** | 不同阵法的"速度加成"差异 → 影响出手顺序 |
| **职业差异** | 不同阵法对应不同职业(输出阵 vs 防御阵)|

## 2.5 阵法相克

### 2.5.1 阵法相克的本质

阵法相克 = 两个阵法相遇时的"属性修正":
- 己方阵 vs 敌方阵 → 查到相克表 → 应用"受伤害减少 / 对方受伤害增加"

### 2.5.2 相克的常见模式

| 模式 | 描述 |
|------|------|
| **环形相克** | A 克 B,B 克 C,C 克 A(像石头剪刀布)|
| **线性相克** | A → B → C → D(一条链)|
| **N×N 矩阵** | N 种阵法,每两两之间有相克关系 |

### 2.5.3 相克的设计意图

- 增加"知己知彼"的决策维度
- 避免"无敌阵法"(每个阵法都有克制)
- 增加"针对性布阵"(知道敌方阵法 → 选克阵)

## 2.6 阵法数据结构

```csharp
public class FormationData {
    public int FormationId;       // 唯一 id
    public string Name;
    public FormationBonus[] Bonuses;  // 5 个位置各自的加成
    public Dictionary<int, float> CounterTable;  // vs 其他阵法 的相克系数
}

public class FormationBonus {
    public int Position;       // 1-5 号位
    public NumericSubType SubType; // 加成 subType
    public int NumericId;       // 加成哪个属性
    public int Value;          // 加成值(百分比 10000 为基数)
}

public class FormationInstance {
    public FormationData Definition;
    public int Level;          // 1-5 级
    public int Position;       // 当前激活的站位(队长设置)
}
```

## 2.7 阵法的实现要点

### 2.7.1 用 `NumericComponent.Modify` 实现

```csharp
// 阵法加成 = 改 FinalPercent
formation.Modify(NumericId.物伤, NumericSubType.FinalPercent, +2000);  // +20%
// 或
formation.Modify(NumericId.法伤, NumericSubType.FinalPercent, +2000);
// 或
formation.Modify(NumericId.速度, NumericSubType.FinalPercent, +1500);  // +15%
```

### 2.7.2 不需要新建 `FormationComponent`

**简化方案**:
- 阵法加成本质是 `FinalPercent` 调整
- 用 `NumericComponent.Subscribe` 监听"阵法激活"事件 → 自动应用
- **不需要**新建 `FormationComponent`

### 2.7.3 阵法生效条件

- 5 人队伍才生效
- 队长在战斗前"选择阵形"
- 进入战斗后按阵形站位

### 2.7.4 阵法等级

- 阵法可升级(1-5 级)
- 1 级阵法 = 满级阵法的 60% 加成(常见)
- 升级后属性按比例提高

## 2.8 阵法 vs 队伍规模

| 队伍规模 | 阵法生效? |
|---------|---------|
| 1 人 | 否 |
| 2-4 人 | 否 |
| 5 人 | **是**(生效)|
| 5 人 + 召唤兽 | 是(5 人 + 召唤兽独立)|
| 6+ 人 | 不允许(常见 5 人上限)|

## 2.9 阵法的常见反模式

### 2.9.1 单一最强阵法

**问题**:某个阵法太强,所有玩家都用它。

**修正**:
- 相克表平衡(每个阵法都有克制)
- 加成范围调整(削弱最强阵法)
- 增加"针对性布阵"机制(知道敌方阵法 → 选克阵)

### 2.9.2 阵法选择无意义

**问题**:玩家不在意阵法 → 失去设计意图。

**修正**:
- 阵法加成大到"必须选"
- 增加"阵法选择"反馈(UI 醒目)
- 增加"阵法相克"的实战影响

### 2.9.3 阵法 vs 站位混淆

**问题**:玩家搞不清"阵法"和"站位"。

**修正**:
- 阵法 = 队伍层级(5 人都受影响)
- 站位 = 单人位置(1 个人受影响)
- UI 明确区分"队伍阵法"和"个人站位"

## 2.10 阵法系统的常见边界

### 2.10.1 阵法 vs 站位 vs 阵位

| 概念 | 描述 |
|------|------|
| **阵法** | 一组人的"队伍策略"(全队加成)|
| **站位** | 单人在队伍中的位置(影响被谁优先打)|
| **阵位** | 同"站位" |

### 2.10.2 阵法 vs 召唤兽站位

- 5 人 + 召唤兽 → 召唤兽通常占"单独位置"
- 阵法加成只作用于 5 人玩家
- 召唤兽不受阵法加成(独立体系)

### 2.10.3 阵法 vs 副本 / PvP

- 副本阵法 = PvE(常见限制:不能相克)
- PvP 阵法 = 玩家对玩家(常见限制:相克生效)

## 本章沉淀的设计模式

沉淀的不是具体某款 RPG 的召唤兽 / 阵法数值,**而是召唤兽 + 阵法系统的设计骨架**:

### 召唤兽:
1. **4 大要素** —— 资质 / 成长 / 技能 / 内丹。
2. **参战副本模式** —— 本体持久化,战斗副本临时。
3. **召唤兽 AI** —— 行为树(残血 / 保护主人 / 默认)。
4. **召唤兽死亡规则** —— 当回合不可出战,战斗结束复活。

### 阵法:
1. **阵法 = 站位 + 属性加成 + 相克** —— 队伍层级的策略。
2. **阵法 = 改 `FinalPercent`** —— 不需要新建 `FormationComponent`。
3. **阵法相克 N×N 矩阵** —— 增加"知己知彼"的决策维度。
4. **阵法生效条件** —— 5 人队 + 队长设置。
5. **阵法等级 1-5** —— 1 级 = 满级 60%(常见比例)。

具体数值(混沌兽攻资最高 1440、最低 1200;天覆阵全体物法伤害+20% 速度-20%)是梦幻西游的数值,**不**沉淀到这里。
