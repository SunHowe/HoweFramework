# 03 · 属性体系

> 本章沉淀回合制 RPG 的属性设计 —— 不含具体游戏数值,只沉淀"为什么这样设计"。

---

## 一、为什么需要属性系统

### 1.1 属性的本质

属性是"角色数据层的可计算参数"。**所有战斗数值都源自属性组合**:
- HP 取决于"基础 HP + 装备 + 等级 + buff"
- 物伤取决于"基础物伤 + 武器 + 力量 + buff"
- 暴击率取决于"基础暴击 + 装备词条 + 状态"

### 1.2 属性系统的设计目标

1. **可组合** —— 装备 / 状态 / 等级 都能叠加,最终值可预测。
2. **可订阅** —— 装备变 → HP 上限自动重算 → UI 自动刷新。
3. **可配置** —— 策划改 `.json` / `.asset` 改公式,不改代码。
4. **可扩展** —— 新增属性不破坏现有数据。

### 1.3 用框架 `NumericComponent` 实现

`NumericComponent` 是 RPG 属性系统的标准实现:
- 支持 `(id, subType)` 复合键(id = 属性,subType = 基础/百分比/固定加值/最终)
- 支持 `Subscribe(id, subType, handler)` 监听变化
- 自动计算 `Final`(链式公式)
- 与 `ResourceComponent.BindNumericMax` 联动(HP 上限绑基础 HP)

## 二、基础属性(8 项)

### 2.1 标准基础属性集合(回合制 RPG)

| 属性 | 含义 | 决定什么 |
|------|------|---------|
| **气血 / HP** | 生命 | 归零即倒地 / 死亡 |
| **魔法 / MP** | 法力 | 大部分法术消耗 |
| **物理伤害** | 攻击 | 物理伤害技能 |
| **物理防御** | 防御 | 减少物理伤害 |
| **法术伤害** | 攻击 | 法术伤害技能 |
| **法术防御** | 防御 | 减少法术伤害 |
| **治疗强度** | 治疗 | 提升治疗技能效果 |
| **速度** | 出手顺序 | 决定回合内谁先行动 |

### 2.2 为什么是这 8 项,不是 6 项或 10 项?

**设计意图**:这 8 项是"决策维度"的最小完备集合。

- 少了"治疗强度" → 治疗数值跟法伤绑定 → 失去治疗职业的差异化
- 少了"法术伤害 / 法术防御" → 双路攻击无法实现(详见 §4)
- 少了"速度" → 出手顺序无依据
- 多了"幸运 / 魅力"等 → 数值膨胀但决策空间不增

**结论**:8 项是"决策维度足够,数值膨胀可接受"的平衡点。

### 2.3 高级属性(常见 6 项,部分游戏 7 项)

| 属性 | 含义 |
|------|------|
| **物理暴击率** | 物理暴击概率 |
| **法术暴击率** | 法术暴击概率 |
| **抗物理暴击** | 降低受到物理暴击的概率 |
| **抗法术暴击** | 降低受到法术暴击的概率 |
| **封印命中** | 封印技能命中率 |
| **抵抗封印** | 被封印命中率降低 |

**为什么"高级"**:这 6 项不影响"基本循环",只影响"特殊循环"(暴击 / 封印)。

## 三、双路攻击:物理 vs 法术(关键设计)

### 3.1 为什么不能合并成单一"攻击力"

**反例**(单路攻击):
- "攻击力"统一,物伤 / 法伤 技能共用一个数值。
- 防御统一,物理 / 法术防御共用一个数值。

**问题**:
- 装备差异化失效 —— "加物伤"和"加法伤"是同一个属性。
- 职业差异化失效 —— 物理输出和法术输出共用装备 / 技能,职业定位模糊。
- 战斗平衡失效 —— 物理 / 法术谁强谁弱无法调,只能动一个数值。

### 3.2 双路的设计意图

| 维度 | 物理路 | 法术路 |
|------|--------|--------|
| 攻击属性 | 物伤 | 法伤 |
| 防御属性 | 物防 | 法防 |
| 主要使用者 | 战士 / 刺客 / 弓手 | 法师 / 道士 |
| 抗性 | 物抗 / 抗物理暴击 | 法抗 / 抗法术暴击 |
| 穿透 | 物穿(忽视目标物防)| 法穿(忽视目标法防)|

### 3.3 双路的"反设计"案例

**错误示范**:某游戏为简化合并"攻击力"和"防御力",后续想做"物法平衡"时发现改不动 → 数值系统大改 → 数据迁移 → 玩家流失。

### 3.4 治疗强度为什么独立

**反例**(治疗强度跟法伤绑定):
- 法系职业的"治疗"技能实际伤害也很高 → 法系职业 = 治疗 + 输出 = 职业定位崩溃。

**设计意图**:
- 治疗是"增益"行为,不应跟"输出"行为混在一起。
- 治疗强度独立的 RPG,治疗职业(化生寺 / 白魔导士 / 僧侣)能纯辅助,不兼任输出。

## 四、属性的链式公式

### 4.1 标准公式

```
Final = (Basic * (1 + BasicPercent/10000) + BasicConstAdd)
        * (1 + FinalPercent/10000) + FinalConstAdd
```

**核心**:**单一事实来源**(single source of truth)。

### 4.2 子类型枚举(以 `Doc/gameplay/numeric-system.md` 为准)

| subType | 值 | 用途 |
|---------|-----|------|
| `Basic` | 0 | 基础值(等级 / 装备 / 状态加成都累加到这个) |
| `BasicPercent` | 1 | 基础百分比(10000 为基数,2000 = 20%)|
| `BasicConstAdd` | 2 | 基础固定加值(被 BasicPercent 缩放)|
| `FinalPercent` | 3 | 最终百分比(**不被** BasicPercent 缩放)|
| `FinalConstAdd` | 4 | 最终固定加值(**不被**任何百分比影响)|
| `Final` | 5 | 只读计算结果(不要手动 Set)|

### 4.3 实战:装备加值如何映射

| 装备效果 | 用什么 subType |
|---------|---------------|
| +10% 物伤 | `FinalPercent = 1000`(10%)|
| +20 固定物伤 | `FinalConstAdd = 20` |
| +50 基础 HP | `BasicConstAdd = 50`(等级加成会缩放它)|
| +15% 基础 HP | `BasicPercent = 1500` |
| +30% 最终 HP | `FinalPercent = 3000` |

### 4.4 公式的常见误用

| 误用 | 正确做法 |
|------|---------|
| 写 `BasicPercent = 0.2` | 写 `BasicPercent = 2000`(10000 为基数)|
| 把"+20 物伤"用 `Basic` 加 | 用 `FinalConstAdd`(不被百分比缩放)|
| 混淆 `Basic` 和 `BasicConstAdd` | `Basic` 是数值基础,`BasicConstAdd` 是固定加值 |

## 五、HP / MP 上限的链式反应

### 5.1 链式机制

```
NumericComponent.(气血, Basic) = 100
ResourceComponent.(HP, Max) = NumericComponent.(气血, Final) = 100
ResourceComponent.(HP, Current) = 100(满血)

装备 +20 气血:
    NumericComponent.(气血, BasicConstAdd) = 20
    → NumericComponent.(气血, Final) = 120(自动重算)
    → ResourceComponent.(HP, Max) = 120(订阅触发)
    → 如果 Current == OldMax:Current = NewMax(满血自动填满)
```

### 5.2 关键 API

```csharp
// 1. 绑定上限
resourceComponent.BindNumericMax(
    resourceId: ResourceId.HP,
    maxNumericId: NumericId.气血,
    numericComponent: numericComponent);

// 2. 装备变 → 触发链式
numericComponent.Modify(NumericId.气血, NumericSubType.BasicConstAdd, +20);
// 内部自动重算 Final → 通知 ResourceComponent 改 Max → 满血自动填满
```

### 5.3 链式的好处

| 场景 | 链式反应 |
|------|---------|
| 装备变 | HP 上限自动重算 |
| 满血时装备变 | HP 上限变,Current 自动填满(满血) |
| 等级提升 | 基础属性增长 → HP / MP 上限自动重算 |
| 阵法加成 | FinalPercent 改 → 所有属性自动重算 |
| buff 加属性 | FinalConstAdd 改 → 持续属性自动重算 |

### 5.4 反例:不用链式

```csharp
// ❌ 反例:每次装备变都要手动同步 HP 上限
void OnEquipmentChanged(Equipment newEquip) {
    var newMaxHp = ComputeMaxHp();
    resourceComponent.SetMax(ResourceId.HP, newMaxHp);
    if (resourceComponent.Get(ResourceId.HP) == oldMaxHp) {
        resourceComponent.Set(ResourceId.HP, newMaxHp);  // 满血自动填满
    }
}
```

**问题**:
- 每个改属性的地方都要记得手动同步。
- 一处遗漏 → 数据不一致。
- 难以测试(每条路径都要单独测)。

**链式**:**用 `Subscribe` 让框架自动推**。

## 六、属性成长曲线

### 6.1 成长的三个维度

| 维度 | 常见模型 | 设计意图 |
|------|---------|---------|
| **等级成长** | 线性 / 指数 / 对数 | 数值膨胀速度 |
| **装备成长** | 固定词条 + 随机词条 | 数值差异 + 玩家动力 |
| **状态成长** | 加成 / 减益 | 战斗内短期变化 |

### 6.2 等级成长曲线

| 曲线 | 公式特点 | 体验 |
|------|---------|------|
| **线性** | `属性 = a + b * lv` | 前期太弱,后期太强 |
| **指数** | `属性 = a * (1 + b)^lv` | 后期数值膨胀,平衡难做 |
| **对数** | `属性 = a + b * ln(lv)` | 后期增长慢,平衡好做 |
| **分段线性** | 每 10 级一段 | 平衡 + 阶段感 |

### 6.3 装备成长曲线(常见模式)

| 模式 | 描述 | 体验 |
|------|------|------|
| **固定词条** | 装备定死属性 | 简单 |
| **随机词条** | 装备生成时随机 1-3 个词条 | 多样性 |
| **稀有度** | 白 / 绿 / 蓝 / 紫 / 橙 / 红 | 商业化 + 收集 |
| **强化** | 装备可升级,升满级有特殊效果 | 长期养成 |

## 七、属性数据存储

### 7.1 存储格式

| 格式 | 适用 | 优劣 |
|------|------|------|
| **ScriptableObject** | 静态数据(技能表 / 装备表) | 编辑器友好,无法运行时改 |
| **JSON / XML** | 配置表 | 灵活,需要解析 |
| **二进制(Luban 等)| 大型数据表 | 高效,需要打包工具 |
| **DataTable 模块** | HoweFramework 的 Luban 集成 | 框架默认,4 种加载模式 |

### 7.2 推荐的存储分层

```
静态配置(ScriptableObject / DataTable)
    ↓ 读
实体数据(NumericComponent + ResourceComponent)
    ↓ 订阅
UI 状态(订阅数值变化)
    ↓ 持久化
存档(序列化到 JSON / 二进制)
```

### 7.3 存档序列化

- 基础属性 → 用 `TakeSnapshot()` / `RestoreSnapshot(snap)`
- 资源 → 单独存(Current / Max)
- 状态 → 单独存(stateId + 剩余回合 + provider)
- 召唤兽 → 单独存(单独数据类)

## 八、属性平衡

### 8.1 平衡的三个维度

| 维度 | 设计目标 | 平衡手段 |
|------|---------|---------|
| **职业平衡** | 各职业有差异化定位 | 装备 / 技能 / 属性差异化 |
| **战斗节奏** | 不让一招秒 / 一回合爆 | 伤害公式 + 防御 + 暴击上限 |
| **成长曲线** | 后期不数值膨胀 | 对数成长 + 等级上限 |

### 8.2 平衡的常见模式

| 模式 | 描述 |
|------|------|
| **剪刀差** | 高攻击同时低防御,低攻击同时高防御 |
| **三角平衡** | 三职业互克(A 强 B,B 强 C,C 强 A)|
| **角色分工** | 坦克 / 输出 / 辅助 / 控制,各自不可替代 |
| **装备词条** | 装备提供差异化空间 |

### 8.3 平衡的"反模式"

| 反模式 | 问题 |
|--------|------|
| **单属性膨胀** | 一条属性决定一切,其他属性无意义 |
| **数值膨胀** | 后期属性涨 100 倍,前后期体验断裂 |
| **职业同质化** | 所有职业共用同一属性 / 装备,失去差异化 |
| **属性过剩** | 玩家超出属性上限,数值无意义 |

## 九、属性系统的常见边界

### 9.1 速度属性的特殊性

- 速度影响**所有排序**,而不是单一技能。
- 速度被改 → 排序重排。
- 速度 = 0 / 负数 → 单位不能行动。

### 9.2 HP 上限的边界

- HP 上限 = 0 → 单位立即判负(不能行动)。
- HP Current > Max → 满血时自动填满机制。
- HP Current < 0 → 倒地 / 死亡(详见 `08-rules-and-edge-cases.md`)。

### 9.3 治疗强度的边界

- 治疗强度 = 0 → 治疗技能 = 基础值。
- 治疗强度 < 0 → 治疗变成伤害(罕见设计)。

### 9.4 暴击率的边界

- 暴击率 ≥ 100% → 每下都暴击。
- 暴击率 ≤ 0 → 从不暴击。
- 抗暴属性 → 削减对方暴击率,**不**让对方暴击率变负。

## 十、本章沉淀的设计模式

沉淀的不是具体某款 RPG 的属性数值,**而是属性系统的设计骨架**:

1. **8 项基础属性 + 6 项高级属性** —— 决策维度的最小完备集合。
2. **双路攻击(物理 / 法术)独立** —— 装备 / 职业 / 平衡的基础。
3. **治疗强度独立** —— 治疗职业不被输出职业吞并。
4. **链式公式** —— `Basic / BasicPercent / BasicConstAdd / FinalPercent / FinalConstAdd / Final`。
5. **HP / MP 上限自动重算** —— `BindNumericMax` 订阅链式。
6. **百分比 10000 为基数**(2000 = 20%) —— 不是小数。
7. **成长曲线** —— 线性 / 指数 / 对数 / 分段线性。
8. **平衡的反模式** —— 单属性膨胀 / 数值膨胀 / 职业同质化 / 属性过剩。

具体数值(基础属性上限 9999、暴击上限 100%、阵法加成 20%)是梦幻西游的数值,**不**沉淀到这里。
