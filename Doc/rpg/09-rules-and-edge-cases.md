# 09 · 通用规则与边界情况

> 本章沉淀回合制 RPG 的通用规则 + 边界情况 —— 不含具体游戏数值,只沉淀"为什么这样处理"。
>
> **架构澄清**:本章涉及的"中毒 / 灼烧 / 隐身 / 倒地 / 死亡"等持续状态,都是 **Provider**(典型 = buff)。详见 [`05-state-component.md`](05-state-component.md) 和 [`06-buff-system.md`](06-buff-system.md)。

---

## 一、回合制 RPG 的核心规则

### 1.1 战斗规则(常见)

| 规则 | 描述 | 设计意图 |
|------|------|---------|
| **每回合每单位 1 次行动** | 每个单位每回合最多出手 1 次 | 回合节奏 |
| **按速度排序** | 速度快的先出手 | 速度属性的意义 |
| **同速 tiebreak** | 同速时按站位 / 阵营 / 随机 | 避免无限循环 |
| **30 秒指令窗口** | 玩家 N 秒下完指令 | 半自动体验 |
| **超时自动判定** | 默认物理攻击 | 不浪费回合 |
| **150 回合上限** | 战斗最多 N 回合 | 防止无限战斗 |
| **明/暗/强制战斗** | 战斗触发方式 | 战斗外流程 |

### 1.2 队伍规则

| 规则 | 描述 |
|------|------|
| **队伍上限** | 常见 5 人(部分游戏 4-6 人) |
| **每人有 1 宠** | 玩家可携带 1 只召唤兽参战 |
| **NPC 队伍上限** | 常见 14 个怪(部分游戏 6-20 个)|
| **阵法生效条件** | 5 人队才生效(详见 [`07-pet-and-formation.md`](07-pet-and-formation.md))|

### 1.3 召唤兽规则

| 规则 | 描述 |
|------|------|
| **参战人数** | 玩家每人 1 宠 |
| **召唤** | 进入战斗时召唤 / 战斗中召唤 |
| **死亡** | 当回合不可出战(用 `DeathFlag` Provider,见 [`06-buff-system.md`](06-buff-system.md) §6)|
| **复活** | 战斗结束 / 复活技能 |
| **鬼魂系** | 可复活(次数限制,也是 Provider)|

### 1.4 胜负规则

| 规则 | 描述 |
|------|------|
| **判负条件** | 一方所有单位 HP=0 或全部逃跑 |
| **平局** | 达到回合上限(常见 30-150)|
| **胜负结算** | 经验 / 掉落 / 任务进度 / 积分 |

## 二、HP=0 的三种边界(倒地 vs 死亡 vs 鬼魂)

### 2.1 三种状态的区别

| 状态 | 触发 | 行为 | 复活 |
|------|------|------|------|
| **倒地**(玩家) | 玩家 HP=0 | 不能行动,本场可救起 | 复活技能 / 复活药品 |
| **死亡**(NPC / 召唤兽)| NPC HP=0 | 当回合不可再出战 | 一般不可复活 |
| **鬼魂**(特殊 NPC)| 鬼魂系 HP=0 | 可复活(次数限制)| 战斗结束前复活不算死亡 |

### 2.2 三种状态的实现(典型 Provider 模式)

详见 [`06-buff-system.md`](06-buff-system.md) §6 "特殊 buff(倒地 / 死亡 / 鬼魂)"。

```csharp
// 典型实现:每个特殊状态都是一个 Provider
public class DownFlag : IBuffProvider {
    public int StateId => StatusId.Down;
    // ... 详见 06-buff-system.md
}

public class DeathFlag : IBuffProvider {
    public int StateId => StatusId.Dead;
    // ... 详见 06-buff-system.md
}

public class GhostFlag : IBuffProvider {
    public int StateId => StatusId.Ghost;
    // ... 详见 06-buff-system.md
}
```

### 2.3 三种状态的 UI 表现

| 状态 | UI |
|------|-----|
| **倒地** | 单位灰显 + "倒地"字样 |
| **死亡** | 单位消失 + 留下尸体 |
| **鬼魂** | 单位半透明 + 飘忽 |

### 2.4 三种状态的战斗后果

- 一方所有玩家**倒地** → 玩家方判负
- 一方所有 NPC **死亡** → 敌方方判负
- 鬼魂系 NPC 死亡后,还能复活 → 仍然算存活

### 2.5 三种状态的特殊规则

| 规则 | 描述 |
|------|------|
| **倒地时所有状态失效** | 倒地后,角色此前所有 buff / Provider 失效(Provider 自己 `OnRemove`)|
| **战斗结束仍未救起 → 判真死** | 玩家倒地 + 战斗结束未救起 = 真死(扣经验 / 金钱 / 物品) |
| **NPC 死亡当回合不可出战** | 召唤兽 / NPC 死亡后,本回合仍占位置但不出手 |
| **鬼魂系复活次数限制** | 部分游戏的鬼魂系有"复活次数"上限,过了就真死 |

## 三、掉线 / 强退处理

### 3.1 掉线的常见处理

| 处理方式 | 描述 | 适用 |
|---------|------|------|
| **离线托管 AI** | 掉线角色由 AI 托管,继续战斗 | 半自动 RPG |
| **战斗结束判定掉线** | 战斗结束才判定掉线,期间正常战斗 | 弱联机 / 单机 |
| **强制结束战斗** | 掉线 = 退出战斗 → 视为逃跑 | 强联机 |
| **重新连接继续** | 允许重连,继续战斗 | MMO |

### 3.2 掉线 vs 强退

- **掉线**:网络断开 / 客户端崩溃
- **强退**:玩家主动退出(Alt+F4 / 关闭客户端)
- 两者处理方式可能不同(掉线可能重连,强退不会)

### 3.3 掉线的"重连机制"

| 机制 | 描述 |
|------|------|
| **保留战斗状态** | 服务器保留掉线玩家的战斗状态,允许重连 |
| **超时清场** | 掉线 N 分钟后,战斗状态销毁 |
| **战斗内提示** | 通知队友"玩家 X 掉线" |

### 3.4 掉线的常见反模式

| 反模式 | 问题 |
|--------|------|
| **掉线 = 直接判负** | 惩罚过重,玩家体验差 |
| **掉线可以无限重连** | 防作弊失效 |
| **掉线时无 AI 托管** | 队伍无法继续战斗 |

## 四、战斗触发的三种方式

### 4.1 明雷战斗

- 可见 NPC 造型(地图上能看到)
- 主动接触触发
- 玩家可以预判(选择 / 避免)

### 4.2 暗雷战斗

- 野外行走概率遭遇
- 自动切战斗画面
- 玩家无法预判(随机)

### 4.3 强制攻击

- 按 Alt+A 点击目标触发
- PK / 比武 / 决斗属此类
- 玩家主动

### 4.4 三种触发的设计意图

| 触发 | 玩家体验 |
|------|---------|
| **明雷** | "我选择打 / 不打" |
| **暗雷** | "我可能在战斗" |
| **强制** | "我主动打" |

### 4.5 三种触发的实现差异

| 触发 | 实现差异 |
|------|---------|
| **明雷** | NPC 实体化,玩家移动触发进入战斗 |
| **暗雷** | 玩家移动时按概率计算 |
| **强制** | 玩家指令触发,需要双方同意(部分游戏) |

## 五、战斗结束的处理

### 5.1 战斗结束的判定

| 判负条件 | 描述 |
|---------|------|
| **一方所有单位 HP=0** | 立即判负 |
| **一方全部逃跑** | 立即判负 |
| **达到回合上限** | 平局(常见 30-150 回合)|
| **特定条件** | 触发剧情条件(比如 boss 战 HP=10% 自动触发剧情) |

### 5.2 战斗结束的"清理"

```csharp
// 1. 销毁所有战斗实体
foreach (var entity in ctx.AllUnits) {
    GameEntityManager.Instance.DestroyEntity(entity.EntityId);
}

// 2. 关闭战斗 UI
UIModule.Instance.CloseForm(BattleFormId.Battle);
UIModule.Instance.CloseForm(BattleFormId.CommandPanel);

// 3. Dispose Context
ctx.Dispose();
```

### 5.3 战斗结束的"结算"

| 结算项 | 描述 |
|--------|------|
| **经验** | 击杀 NPC 给经验 |
| **掉落** | 击杀 NPC 给物品(掉率表)|
| **任务进度** | 完成任务目标 |
| **积分** | PvP 积分 / 排行 |
| **惩罚** | 死亡惩罚(扣经验 / 金钱) |
| **战利品分配** | 队伍内分配规则(轮流 / 自由 / 拍卖)|

### 5.4 战斗结束的"再来一场"

- 新建一个 `BattleContext`(不复用旧的)
- 老 Context 已被 Dispose,所有状态自动清零
- 玩家按"再来一场"按钮 → 重新进入战斗

## 六、保护机制

### 6.1 保护的本质

保护 = "分担队友受到的物理攻击":
- 玩家手动触发("保护"指令)
- 仅当目标受到物理攻击时触发
- 1 次保护只挡 1 次攻击

### 6.2 保护的常见规则

| 规则 | 描述 |
|------|------|
| **触发条件** | 目标受到物理攻击(法术不触发) |
| **保护者状态** | 保护者未倒地 / 未被封 |
| **伤害处理** | 保护者承担全部物理伤害(部分游戏分担) |
| **保护次数** | 1 次保护只挡 1 次攻击 |
| **保护者选择** | 通常是同队伍 / 优先坦克 |

### 6.3 保护的实现

保护 = 一个 Provider(典型 = `BuffProtecting`),详见 [`06-buff-system.md`](06-buff-system.md)。

```csharp
// 保护 = 创建一个 BuffProtecting Provider
var protecting = m_Context.BuffPool.Acquire<BuffProtecting>(protector, caster, duration=1);
protecting.OnApply(ctx);
// 当目标受到物理攻击时,先检查"是否有 BuffProtecting Provider 在队友身上"
// 如果有,伤害转给保护者
```

### 6.4 保护 vs 防御

| 保护 | 防御 |
|------|------|
| 主动指令(玩家手选)| 主动指令(玩家手选)|
| 队友分担伤害 | 自己承受减伤(常见 50%)|
| 仅挡 1 次 | 持续 1 回合 |
| 物理攻击触发 | 物理 + 法术都减伤 |

### 6.5 保护的常见反模式

| 反模式 | 问题 |
|--------|------|
| **保护无敌** | 保护者无敌 → 队伍失衡 |
| **保护无次数限制** | 保护者反复保护 → 失衡 |
| **保护挡法术** | 设计意图失效 |

## 七、半自动指令的常见边界

### 7.1 指令下完后的"锁定"

- 玩家下完指令 → 锁定(不可改)
- 设计意图:防止"看了动画又改"
- 实现:`Dictionary<EntityId, Command>` 存每个单位的指令,执行前不可改

### 7.2 指令超时

- 默认操作(常见:物理攻击,无目标时随机)
- 设计意图:让玩家觉得"反正输了也比这好"

### 7.3 指令被禁用

某些状态(Provider)可能禁用某些指令:
- 被封(`BuffSeal`)→ 无法放法术
- 沉默(`BuffSilence`)→ 无法放技能
- 嘲讽(`BuffTaunt`)→ 强制只能攻击嘲讽者
- 混乱(`BuffConfusion`)→ 随机选择目标

### 7.4 指令冲突

- 多单位同时给同一目标下"保护" → 取第一个 / 随机
- 多单位同时给同一目标下"复活" → 取第一个 / 随机

## 八、跨回合状态的处理

### 8.1 跨回合持久化

| 状态 | 是否跨回合 |
|------|----------|
| **HP / MP** | 是(战斗内)|
| **愤怒** | 是(战斗内)|
| **buff / Provider** | 持续 N 回合(可能跨回合)|
| **倒计时** | 持续 N 秒(可能跨回合)|
| **复活标记** | 是(到下次复活时间) |

### 8.2 跨回合的"安全引用"

- AI 行为树在第 3 回合选了一个目标,第 8 回合回来还在用同一个引用
- 目标在第 5 回合被销毁 → 裸 `IGameEntity` → `NullReference`
- **必须用 `GameEntityRef`**(自动 null-check)

### 8.3 跨回合的"清理"

- 战斗结束 → 销毁所有战斗实体
- 不需要"重置" — Dispose 整个 Context → 所有自动清零

## 九、录像 / 回放

### 9.1 录像的本质

录像 = "战斗的可重放序列":
- 存 `GameRandomManager.SetSeed(seed)` + 玩家操作序列
- 回放时 `SetSeed(原seed)`,按序列重放
- AI 用同样的随机数序列,行为完全可重放

### 9.2 录像的存什么

| 数据 | 描述 |
|------|------|
| **种子** | `GameRandomManager.CurrentSeed` |
| **玩家操作** | 每个玩家单位每回合的操作序列 |
| **初始状态** | 进入战斗时的实体 / 属性 / 状态 |
| **回合触发** | 回合开始 / 结束事件 |

### 9.3 录像的回放

- 重新创建 `BattleContext`
- 重新加载初始状态
- 重新设置种子
- 按操作序列重放

### 9.4 录像的常见误用

| 误用 | 问题 |
|------|------|
| **自己 `new Random()`** | 不可回放 |
| **存最终结果,不是操作序列** | 无法回放 |
| **种子存错(存了 final seed)** | 回放行为不一致 |

## 十、网络同步(MMORPG)

### 10.1 服务器权威

- 客户端只发"指令"(施法 / 移动)
- 服务端裁决(命中 / 伤害)
- 客户端显示结果(动画 / 数字)

### 10.2 状态同步

- 客户端每帧广播自身状态(HP / MP / 位置)
- 服务端每秒广播全局状态(所有玩家 / NPC)
- 客户端插值(平滑显示)

### 10.3 网络延迟

- 接受 100-300ms 延迟
- 客户端预测(本地先响应,服务端后确认)
- 服务端协调(纠正客户端预测错误)

### 10.4 网络掉线

- 客户端断线 → 服务器保留 N 秒状态
- 重新连接 → 同步当前状态
- 超过 N 秒未连 → 角色视为退出战斗

## 十一、其他常见边界

### 11.1 同名玩家 / 队伍

- 队伍重名 → 加后缀(A 队 / B 队)
- 玩家重名 → 加编号 / 后缀

### 11.2 跨服 / 跨区

- 跨服战 → 临时队伍(原队伍解散)
- 跨服聊天 → 限制频道

### 11.3 经济系统

- 货币类型(金币 / 元宝 / 点券)
- 货币用途(强化 / 购买 / 拍卖)
- 经济平衡(产出 / 消耗)

### 11.4 反作弊

- 服务端权威(所有裁决在服务端)
- 反外挂(客户端加密 / 验证)
- 反作弊检测(异常行为检测)

## 十二、本章沉淀的设计模式

沉淀的不是具体某款 RPG 的通用规则数值,**而是这些规则的"设计骨架"**:

1. **战斗核心规则** —— 每回合 1 次行动 / 按速度排序 / 30s 指令窗口 / 150 回合上限。
2. **HP=0 三态边界** —— 倒地 / 死亡 / 鬼魂(独立处理,典型用 Provider 模式,见 [`06-buff-system.md`](06-buff-system.md) §6)。
3. **掉线 / 强退处理** —— 离线托管 AI / 战斗结束才判掉线。
4. **战斗触发方式** —— 明雷 / 暗雷 / 强制。
5. **战斗结束清理** —— 销毁实体 / 关 UI / Dispose Context。
6. **保护机制** —— 仅挡物理 / 1 次挡 1 次(用 `BuffProtecting` Provider)。
7. **半自动指令边界** —— 锁定 / 超时 / 被禁用(用 Provider 状态禁用指令)。
8. **跨回合持久化** —— 安全引用 `GameEntityRef`。
9. **录像回放** —— 存 seed + 操作序列。
10. **MMORPG 网络** —— 服务端权威 + 状态同步 + 延迟容忍。

具体数值(150 回合上限、5 人队 + 1 宠 vs 14 怪) 是梦幻西游的数值,**不**沉淀到这里。
